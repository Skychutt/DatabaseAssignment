## 一、开场与项目概述（约 1 分钟）

各位老师、同学，大家好。我们小组的项目是 **学生管理系统（DatabaseAssignment）**，一门面向 **管理员、教师、学生** 三种角色的 **Web 信息系统**，后端采用 **Java Servlet + JDBC** 访问 **MySQL** 数据库，前端使用 **JSP + Bootstrap + jQuery** 提供页面与交互。

系统主要解决：班级与学生信息管理、教师与课程管理、学生选课与教师打分、系统公告发布，以及学生个人档案维护等业务，并在首页展示统计数据与图表。

我们选择 **手写 SQL + DAO** 而不是 Hibernate 等 ORM，符合作业对「Java 方案、不推荐复杂 ORM」的要求，结构清晰，便于在答辩中说明每一条 SQL 与表结构的对应关系。

## 二、整体架构与各层职责（约 2–3 分钟）

项目采用经典 **分层架构**，可对应轻量级 **MVC**：

| 层次 | 包/目录 | 负责什么 | 典型文件 |
|------|---------|----------|----------|
| **表现层（View）** | `src/main/webapp` | 用户看到的页面：表单、列表、侧栏导航；少量 JavaScript 做 AJAX 提交与图表 | `login.jsp`、`index.jsp`、`WEB-INF/view/*.jsp` |
| **控制层（Controller）** | `servlet` | 接收 HTTP 请求（URL、参数），调用 Service，转发 JSP 或返回 JSON | `LoginServlet`、`StudentServlet`、`StuCourseServlet` 等 |
| **业务层（Service）** | `service` | 业务规则与校验：能否删除、选课是否满员、是否越权等 | `StudentService`、`StuCourseService`、`NoticeService` |
| **持久层（DAO）** | `dao` | 与数据库打交道：执行 SQL、把结果封装成对象 | `StudentDao`、`CourseDao`、`StuCourseDao` |
| **实体层（Entity）** | `entity` | 与表字段对应的 Java 类（POJO），在层之间传递数据 | `Student`、`Course`、`Notice`、`StudentArchive` |
| **工具与安全** | `utils`、`filter`、`security` | 数据库连接、分页、JSON 响应；登录拦截；角色判断 | `JdbcHelper`、`AuthFilter`、`RoleHelper` |

### 各层「只干自己的事」

- **Servlet**：不写复杂业务，主要做「路由 + 调 Service + 选哪个 JSP」。
- **Service**：不写 SQL，只关心「这样操作合不合理、合法不合法」。
- **Dao**：不写页面逻辑，只关心「怎么查、怎么增删改」。
- **JSP**：不写数据库连接，只展示 `request` 里 Servlet 放进来的数据。

这种分工便于五人小组并行开发：一人偏库表与 SQL，一人 DAO，一人 Service，一人 Servlet，一人 JSP 与联调。

## 三、系统运行原理（请求是怎么走完的）（约 2 分钟）

以 **学生登录后查看选课** 为例，说明一次完整流程：

```
浏览器  →  Tomcat  →  AuthFilter  →  StuCourseServlet  →  StuCourseService  →  StuCourseDao  →  JdbcHelper  →  MySQL
                ↑                                                                              ↓
                └──────────── 查询结果封装为对象 ── 放入 request ── 转发 stu-course-student.jsp ── HTML 返回浏览器
```

**分步说明：**

1. **浏览器** 访问 `http://localhost:8080/项目名/stuCourse`（GET）。
2. **Tomcat** 根据 `@WebServlet("/stuCourse")` 找到 `StuCourseServlet`。
3. **AuthFilter**（过滤器，拦截 `/*`）先执行：若 Session 里没有 `role`，重定向到 `login.jsp`；已登录则放行。
4. **Servlet** 从 Session 取出当前学生学号，调用 `StuCourseService` 查询已选课程、可选课程分页数据。
5. **Service** 校验参数，调用 **Dao**；Dao 通过 **JdbcHelper** 执行 `SELECT ...`，把 `ResultSet` 转成 `List` 或 `PagerVO`。
6. Servlet 将结果 `setAttribute` 到 `request`，**转发**到 `stu-course-student.jsp`。
7. JSP 渲染 HTML；页面内 **jQuery** 可对「选课」「退选」发 **POST**，Servlet 返回 **JSON**（`ApiResult`），浏览器弹窗提示后刷新。

**POST + JSON 的典型场景**（增删改）：登录、保存学生、发布公告、选课、打分等——页面不整页刷新提交表单，而是用 `$.post(..., function(res){ ... }, "json")`，Servlet 设置 `Content-Type: application/json` 并输出 `{"success":true,"message":"..."}`。

**Session 的作用**：登录成功后在 Session 中保存 `role`（admin/teacher/student）和 `user`（当前用户对象），后续请求靠 Session 识别身份，Filter 与各 Servlet 中的 `RoleHelper` 做权限控制。

## 四、功能模块与角色权限（约 1.5 分钟）

| 角色 | 主要功能 |
|------|----------|
| **管理员** | 教师管理；课程/班级/学生维护；选课总览与移除；公告发布/修改/删除；首页统计 |
| **教师** | 查看班级、学生名册；管理本人课程；对选课学生**打分**；查看公告、通讯录 |
| **学生** | 查看班级与学生信息（只读）；浏览课程；**选课/退选**；维护**个人档案**；查看公告与个人资料 |

**权限实现方式：**

- 全局：`AuthFilter` 要求已登录。
- 细粒度：各 Servlet 的 `doGet`/`doPost` 中用 `RoleHelper.isAdmin()`、`isTeacher()`、`isStudent()` 判断；JSP 用 `canManageXxx` 隐藏无权限按钮。

## 五、数据库设计相关知识（约 2–3 分钟）

数据库名：**`stu_manage`**。核心业务表及关系如下。

### 5.1 主要数据表

| 表名 | 说明 | 主键 |
|------|------|------|
| `tb_admin` | 管理员账号 | `username` |
| `tb_teacher` | 教师 | `tno`（工号） |
| `tb_student` | 学生 | `sno`（学号） |
| `tb_clazz` | 班级 | `clazzno` |
| `tb_course` | 课程 | `cno` |
| `tb_stu_course` | 选课记录（含成绩、评价） | 复合主键 `(cno, sno)` |
| `tb_notice` | 系统公告 | `id`（自增） |
| `tb_student_archive` | 学生个人档案 | `archive_id` |

### 5.2 外键与完整性（体现「数据库课」内容）

- `tb_student.clazzno` → `tb_clazz.clazzno`：学生归属班级。
- `tb_course.tno` → `tb_teacher.tno`：课程由教师授课。
- `tb_stu_course.cno` → `tb_course.cno`，`tb_stu_course.sno` → `tb_student.sno`：选课联系课程与学生（**多对多** 用中间表实现）。
- `tb_notice.publisher` → `tb_admin.username`：公告发布人必须是管理员。
- `tb_student_archive.sno` → `tb_student.sno`：一名学生对应一份档案（`sno` 唯一）。

**设计要点：**

- 使用 **主键、外键、唯一约束**（如身份证号 `id_card` 唯一）保证数据一致。
- `tb_stu_course` 使用 **复合主键**，防止同一学生重复选同一门课。
- 课程表 `count` 字段与选课人数联动，体现业务与数据的配合。

### 5.3 从概念模型到关系模型（答辩可提）

- **实体**：学生、教师、课程、班级、公告、档案等。
- **联系**：授课（教师–课程）、选课（学生–课程）、所属（学生–班级）、发布（管理员–公告）等。
- **实现**：用外键把联系落在表结构上；查询时用 **JOIN**（或分步查询）组装 `StuCourseView` 等展示对象。

### 5.4 本项目如何访问数据库（JDBC）

未使用 Hibernate，流程为：

1. `JdbcHelper` 加载 MySQL 驱动，建立 `Connection`。
2. 使用 `PreparedStatement` 执行带 `?` 占位符的 SQL，防止简单注入。
3. `executeQuery` 得到 `ResultSet`，在 Dao 中逐行 `rs.getString("列名")` 填入 Entity。
4. `executeUpdate` 执行 INSERT/UPDATE/DELETE。
5. 操作结束后 `closeDB()` 释放连接。

**与 ORM 对比（若被问到）**：我们是「SQL 可见、可控」；ORM 则用对象 `persist()` 由框架生成 SQL。作业鼓励我们这种 JDBC + DAO 方式。

## 七、相关知识点小结（面试/问答准备）

1. **Servlet 生命周期**：`init` → 多次 `service`（`doGet`/`doPost`）→ `destroy`；由 Tomcat 容器管理。
2. **转发 vs 重定向**：`forward` 服务器内部转 JSP，地址栏不变；`sendRedirect` 浏览器重新请求（如未登录跳登录页）。
3. **Filter 链**：请求先经过 Filter，再到达 Servlet；我们用它做统一登录校验。
4. **Maven**：管理依赖（Servlet API、MySQL 驱动）与 war 打包。
5. **分页**：`PagerVO` 封装 `current`、`size`、`total`、`list`，Dao 中 `LIMIT offset, size`。
6. **事务**：当前项目以单表或简单操作为主；若扩展可对 `Connection.setAutoCommit(false)` 做选课+更新人数等事务（可作改进点）。
7. **前端 JS 的定位**：仅辅助交互；**业务与数据在 Java 层**，符合作业「Java-based solution」。

## 八、演示建议顺序（约 5 分钟现场 Demo）

1. 打开 `login.jsp`，展示验证码与三种角色登录。
2. **管理员**：首页统计图 → 教师/课程/学生列表 → 发布一条公告。
3. **教师**：选课与打分页，修改一名学生成绩并保存。
4. **学生**：我的选课（选课或退选）→ 个人档案（新建或编辑）→ 查看公告（只读）。
5. 简述：未登录直接访问 `/student` 会被 Filter 拦到登录页。

## 九、已知局限与可改进点（体现思考，可选讲）

- 密码为明文存储，生产环境应使用哈希（如 BCrypt）。
- 部分 Dao 动态拼接 `WHERE` 需注意 SQL 注入，建议全面改为参数化。
- SQL 备份脚本建议提供精简版，仅含 `stu_manage` 业务表。
- 可选扩展：事务、连接池（如 HikariCP）、日志框架。
